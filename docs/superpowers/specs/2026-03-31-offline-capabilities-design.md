# Offline Capabilities Design

**Date:** 2026-03-31
**Status:** Draft
**Approach:** Workbox (Service Worker) + Dexie.js (IndexedDB) + Custom Sync Engine

## Overview

BirdhouseMapper is used in the field, often away from reliable connectivity. This design adds full offline support: background data caching, offline reads and writes for all operations, a mutation queue with automatic sync on reconnect, map tile pre-downloading, and PWA installability. The experience should be seamless — the app works the same whether online or offline, with no visible offline indicators.

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Conflict resolution | Last-write-wins (timestamp) | Simpler sync engine; acceptable for field use |
| Offline editing scope | All operations | Full offline editing including admin |
| Tile caching | Browse-cache + pre-download per property | Ensures maps work in the field |
| Photo strategy | Offline-captured photos stored locally, uploaded on sync; no pre-download of existing photos | Keeps storage manageable |
| Offline UX | Seamless, no visible indicators | App should just work |
| Data download | Auto-cache current property + manual download for others | Prepared for field work without manual steps |
| Auth offline | Full use without valid token; re-auth before sync | Field workers shouldn't be locked out |

## 1. Local Database (Dexie/IndexedDB)

### Cached Tables (mirroring Supabase)

| Local Table | Source | Purpose |
|---|---|---|
| `items` | `items` | All items for cached properties |
| `item_types` | `item_types` | Item type definitions (icons, colors) |
| `custom_fields` | `custom_fields` | Field definitions for item types |
| `item_updates` | `item_updates` | Observation logs |
| `update_types` | `update_types` | Update type definitions |
| `photos` | `photos` | Photo metadata (not blobs) |
| `entities` | `entities` | Species, volunteers, etc. |
| `entity_types` | `entity_types` | Entity type definitions |
| `geo_layers` | `geo_layers` | GeoJSON overlay data |
| `properties` | `properties` | Property config, map defaults |
| `orgs` | `orgs` | Org config, theme |
| `roles` | `roles` | Role definitions + permissions JSON |
| `org_memberships` | `org_memberships` | Current user's membership/role |

### Offline-Only Tables

| Table | Purpose |
|---|---|
| `mutation_queue` | Pending offline changes (operation, table, payload, timestamp, status) |
| `photo_blobs` | Locally captured photos as Blobs, linked to mutation_queue entries |
| `sync_metadata` | Last sync timestamp per table per property, cache status |
| `tile_cache_metadata` | Track which tile regions/zoom levels are cached |

### Key Design Points

- Each cached record keeps its Supabase UUID as primary key (no ID mapping).
- A `_synced_at` field on each record tracks freshness.
- New records created offline get a client-generated UUID via `crypto.randomUUID()` — Supabase accepts UUID inserts.
- All tables indexed by `org_id` and `property_id` for fast scoped queries.

## 2. Service Worker & Caching Strategy (Workbox)

### Layer 1: App Shell (Precaching)

Workbox precaches the Next.js build output — JS bundles, CSS, HTML shells. The app loads instantly with no connection. Managed by Serwist (`@serwist/next`) integrated into the Next.js build pipeline.

### Layer 2: API/Data Responses (Fallback Safety Net)

After the data access refactor (Section 4), components read/write through the offline store, not Supabase directly. However, the service worker also intercepts Supabase REST API calls as a safety net for any code paths that still hit Supabase directly (e.g., third-party integrations, missed refactors):
- **Online:** Requests pass through to Supabase normally.
- **Offline:** Intercepted requests return a structured offline error, prompting the caller to fall back to local data.

The primary data path is always: component → `useOfflineStore()` → IndexedDB. The service worker intercept is defense-in-depth, not the main offline mechanism.

### Layer 3: Map Tiles (Cache-First)

Tile requests (e.g., `tile.openstreetmap.org`) use a **Cache-First** strategy:
- Tiles are immutable at a given URL — cached tiles are always valid.
- `ExpirationPlugin` caps tile cache at a configurable size (default 500MB) with LRU eviction.
- Pre-download: a background task fetches tiles for a property's bounding box at zoom levels 13-17.

### Layer 4: Photos (Network-Only + Local Capture)

- Existing photos from Supabase Storage: network-only, no pre-download.
- Photos captured offline: stored as Blobs in IndexedDB (`photo_blobs` table), uploaded during sync.

### Offline Detection

- `navigator.onLine` + `online`/`offline` events for instant detection.
- Supplemented by a lightweight ping to Supabase on a timer (catches "connected but no internet" edge cases).
- Exposed via `useNetworkStatus()` React hook so components can adapt behavior without visible UI changes.

## 3. Mutation Queue & Sync Engine

### Unified Write Path

All mutations flow through the local database first, regardless of connectivity:

1. User action writes to IndexedDB immediately (instant UI feedback).
2. A mutation record is enqueued in `mutation_queue`.
3. If online, the sync engine processes the queue immediately.
4. If offline, the queue accumulates and drains when connectivity returns.

### Mutation Queue Record Schema

```typescript
interface MutationRecord {
  id: string;              // UUID
  table: string;           // e.g., 'items', 'item_updates'
  operation: 'insert' | 'update' | 'delete';
  record_id: string;       // UUID of the affected record
  payload: object;         // Full record for insert, changed fields for update
  org_id: string;
  property_id: string;
  created_at: number;      // Timestamp for last-write-wins ordering
  status: 'pending' | 'in_flight' | 'failed' | 'completed';
  retry_count: number;
  error: string | null;
}
```

### Outbound Sync (Client to Server)

- Processes queue in FIFO order (oldest first).
- Marks each entry `in_flight` before attempting.
- On success: marks `completed`, removes after confirmation.
- On failure: increments `retry_count`, marks `failed`, retries with exponential backoff (max 5 retries).
- Auth failure (401/403): pauses sync, triggers re-authentication prompt, resumes after.
- Photo uploads: `photo_blobs` entries are uploaded to Supabase Storage first, then the `photos` table mutation fires with the resulting `storage_path`.

### Inbound Sync (Server to Client)

When coming back online or on a periodic interval:
- For each cached property, fetch records with `updated_at > last_sync_timestamp`.
- Upsert into IndexedDB.
- Last-write-wins: if a local pending mutation conflicts with a newer server record, the server version wins and the local mutation is discarded from the queue.

### Sync Trigger Points

- Connectivity restored (`online` event).
- App foregrounded / tab becomes visible (`visibilitychange` event).
- Periodic poll every 5 minutes while online.
- Manual pull-to-refresh on list/map views.

## 4. Data Access Layer Refactor

### Problem

Components currently call `createClient()` and query Supabase directly. For offline support, an abstraction layer routes reads/writes through IndexedDB.

### Solution: Offline Store

Components call `useOfflineStore()` instead of Supabase directly.

**Before:**
```typescript
const supabase = createClient();
const { data } = await supabase.from('items').select('*').eq('property_id', pid);
```

**After:**
```typescript
const { getItems } = useOfflineStore();
const items = await getItems(propertyId);
```

Under the hood, `getItems()` reads from IndexedDB. The sync engine keeps IndexedDB fresh from Supabase in the background. Writes go through `store.ts` which writes locally and enqueues mutations.

### Server Actions Refactor

Current server actions (`'use server'`) cannot run offline. The approach:
- Extract validation/business logic from server actions into shared functions.
- Client-side calls go through the offline store (local write + queue).
- Server actions remain for server-only operations (cron jobs, admin bulk ops needing service role).
- The sync engine replays mutations as REST calls to Supabase directly (not through server actions).

### Key Principle

Components never know or care whether they're online or offline. The data access layer handles everything transparently.

## 5. Background Data Download & Cache Management

### Auto-Caching (Current Property)

When a user navigates to a property, the sync engine silently downloads all data for that property:
- Items, item types, custom fields, updates, update types, entities, entity types, geo layers, photo metadata.
- Uses a Web Worker to avoid blocking the UI thread.
- Progress tracked in `sync_metadata` — if interrupted, resumes on next visit.
- Fetches in batches (500 records at a time) to avoid memory spikes.

### Manual Download (Other Properties)

A settings page or property list shows cache status per property:
- Status: "Cached" / "Not cached" / "Stale (3 days old)".
- "Download for offline" button triggers background sync for that property.
- "Download all org data" button queues all properties in the org.
- "Clear cache" per property to free storage.

### Tile Pre-Download

When a property is cached, tiles are also pre-fetched:
- Uses the property's `map_default_lat`, `map_default_lng`, and a bounding box derived from item locations or property boundary geo layer.
- Downloads zoom levels 13-17 (practical field use range).
- Estimated size shown before download (tile count x ~15KB avg).
- Runs as a low-priority background task, pausable.

### Storage Budget

- IndexedDB limits: typically 50-80% of free disk space (varies by browser).
- `navigator.storage.estimate()` used to show available space in the download UI.
- LRU eviction on tile cache when approaching limits.
- Data cache (items, updates, etc.) is small — likely under 50MB even for large orgs.
- Tiles are the storage concern — the UI shows estimated tile cache size before download.

### Cache Freshness

- `sync_metadata` stores `last_synced_at` per property per table.
- On app open (if online): delta sync — only fetch records newer than last sync.
- Stale threshold: configurable, default 24 hours — after which a background refresh triggers automatically.

## 6. Authentication & Security Offline

### Offline Session

When the app goes offline or detects token expiry while offline:
- The app continues to function — reads from IndexedDB, writes to mutation queue.
- The user's profile, org membership, role, and permissions are cached in IndexedDB at last sync.
- Permission checks run against cached role/permissions JSON locally — same `usePermissions()` hook, backed by local data.

### Re-Authentication on Reconnect

When connectivity returns and the sync engine activates:
1. Attempt to refresh the Supabase session (refresh token may still be valid — 7-day default).
2. If refresh succeeds: sync proceeds normally.
3. If refresh fails (token fully expired): sync paused, user sees a sign-in prompt, sync resumes after authentication.
4. Mutations stay safe in IndexedDB regardless — nothing is lost.

### Security Boundaries

- Cached data is scoped to what the user already had access to (RLS enforced at download time).
- No service-role operations happen client-side.
- If a user's role is revoked while offline, the next sync fails with 403s on queued mutations — those are marked `failed` and the local cache is cleared for that org.
- IndexedDB is origin-scoped by the browser — tenant isolation via subdomain/custom domain is preserved.

### Persistent Storage

On first offline cache, the app calls `navigator.storage.persist()` to prevent the browser from evicting IndexedDB data under storage pressure. Critical on mobile browsers.

## 7. PWA Manifest & Install Experience

### Web App Manifest

Served dynamically via a Next.js API route (`/api/manifest.json`) based on tenant resolution:
- `name`: From org config (e.g., "Audubon BirdhouseMapper").
- `short_name`: Org short name or "BirdhouseMapper".
- `start_url`: `/map` (most useful starting point in the field).
- `display`: `standalone` (no browser chrome).
- `orientation`: `any` (portrait and landscape).
- `theme_color` / `background_color`: From the org's theme config.
- Icons: Generated from org logo or default BirdhouseMapper icon at 192px, 512px, and maskable variants.

### Install Experience

No custom install banner — rely on browser native install prompts. The app meets all PWA installability criteria (manifest + service worker + HTTPS). Safari on iOS uses the share menu for Add to Home Screen.

### Service Worker Scope

- Scope: `/` (entire app).
- Navigation fallback: offline HTML shell loads the app from cache, client-side routing handles everything.

## Out of Scope

- Real-time collaborative editing (Supabase Realtime channels) — not needed for field use.
- Push notifications for sync status — seamless UX means no notifications.
- Offline admin operations that require service-role access (e.g., bypassing RLS).
- Pre-downloading existing photos — only locally captured photos are stored offline.
- Custom install banners or prompts.
